## Abstract 

Deploying natural language search systems presents a critical cold-start challenge: no real user queries to learn linguistic patterns, and no relevance labels to train ranking models. We present a framework for generating synthetic queries and labels using large language models (LLMs), powering model training and evaluation for Airbnb’s natural language search. For query generation, we combine contrastive listing pairs from booking sessions (ground-ing queries in real platform features) with seed queries from user research (ensuring realistic linguistic patterns). We develop three prompt variants to balance realism and diversity, enabling a cold-to-warm start transition as real user data becomes available. For label generation, we introduce contrastive generation that pro-duces topicality labels by construction, and Virtual Judge (VJ) la-beling for broader coverage. We analyze both query length and attribute distributions comparing our approach against a baseline (contrastive generation without seed data). For query length, the baseline produces verbose queries with KL divergence of 4.95 vs. real users; our seed-guided approach achieves 0.66, a 7.5 × improve-ment. For attribute distributions, our approach achieves the lowest KL divergence for attribute types (0.04)—even outperforming seed queries (0.09)—demonstrating that combining seed guidance with contrastive generation produces the most realistic attribute pat-terns. Experiments show our approach produces harder evaluation examples than the baseline (79% vs. 97% accuracy), providing dis-criminative signal for model improvement. We deploy production pipelines generating synthetic examples daily for embedding-based retrieval and ranking evaluation.

## 1 Introduction 

Airbnb is an online marketplace for short-term stays and experi-ences. Search ranking has been a core focus of optimization [ 6– 9], with models trained on user interactions to predict booking like-lihood. Traditional search relies on structured filters—destination, dates, guest count, amenities—which are precise but limited in ex-pressiveness. Users cannot easily convey preferences like “a quiet retreat for working remotely” through checkboxes. To address this, we are enabling natural language search that complements filters by allowing users to express intent more freely. While e-commerce platforms such as Amazon, Walmart, and Alibaba have long sup-ported natural language queries, travel platforms have tradition-ally relied on structured inputs. The emergence of large language models (LLMs) has made this capability technically feasible, but deploying it presents significant challenges. Launching this new search capability presents a fundamental 

cold-start problem : without historical user data, there is no signal to train or evaluate ranking models. Traditional approaches address this through rule-based systems that collect initial data, or through expensive human labeling campaigns. We take a different approach: using LLMs to generate synthetic training data at scale, enabling model development before real user traffic exists. A key distinction in search ranking is between topicality (whether a listing matches the query’s stated intent) and bookability (whether the user will actually book). Airbnb’s search ranking ultimately optimizes for conversion, predicting which listings users will book. However, booking behavior depends on factors beyond topicality: pricing, availability, photos, reviews, and personal preferences. In a cold-start setting, we have no booking data to learn these pat-terns. Our approach therefore focuses on topicality first: generating synthetic data to train models that understand query-listing rele-vance. As real user data becomes available, these topicality-trained models provide a foundation for warm-start adaptation toward bookability. This paper focuses on the cold-start topicality problem; transitioning to bookability is future work. The cold-start problem manifests as two distinct challenges: (1) No Real Queries —before launch, we cannot observe how users naturally express their needs, what language patterns they use, what attributes they mention, or how verbose or terse they are; without this distribution, we cannot train models that understand natural language intent. (2) No Relevance Labels —even with queries, we lack ground-truth labels indicating which listings are relevant; traditional systems derive relevance from user interactions, but a new search modality has no interaction history. Our framework addresses both challenges while providing a path from cold start to warm start. A key advantage is data effi-ciency : from a small seed set of approximately 500 queries collected through user research surveys, we generate millions of synthetic queries for production-scale model training. Using LLMs, we gen-erate synthetic queries that mimic seed patterns while grounding them in real listing attributes, achieving orders-of-magnitude am-plification from seed to production data. As the system launches

## 2 Related Work 

Our work addresses the cold-start problem in natural language search through synthetic data generation. We review three areas: (1) traditional approaches to cold-start problems and their limitations, (2) synthetic query generation techniques, and (3) methods for generating relevance labels without human annotation. 

## 2.1 Cold-Start Approaches in Search 

Launching new search capabilities without historical data is a com-mon challenge. Rule-based methods use hand-crafted heuris-tics to provide initial rankings—for example, sorting by popularity or recency—and collect user interaction data to train ML models over time. While practical for bootstrapping, these methods cannot capture nuanced user intent and may provide suboptimal initial experiences. Human labeling employs annotators to judge query-document relevance, producing high-quality training data but at significant cost. Crowdsourcing platforms can scale labeling [ 1], but quality control remains challenging, and labeling millions of ex-amples is often prohibitively expensive. Transfer learning adapts models trained on related domains [ 3], but domain shift between source and target can limit effectiveness. Our approach offers a mid-dle path: using LLMs to generate synthetic data that is both scalable (like rule-based methods) and realistic (like human labeling), while requiring only a small seed set of authentic examples. 

## 2.2 Synthetic Query Generation 

Synthetic data generation has emerged as a powerful technique for training NLP systems in low-resource scenarios [ 4, 25 ]. Schick and Schütze [20] demonstrated that LLMs can generate training data for text classification, while Wang et al . [23] showed synthetic data from GPT-3 improves few-shot learning. Self-Instruct [ 24 ]and Stanford Alpaca [ 21 ] introduced methods for models to gener-ate their own instruction-following data. For information retrieval specifically, InPars [ 2 ] and InPars-v2 [ 11 ] use LLMs to generate training queries, while Promptagator [ 3] demonstrates few-shot query generation. Query generation has also been studied for docu-ment expansion [ 17 ] and zero-shot retrieval [ 16 ]. Traditional data-to-text systems [ 5 ] employ multi-stage pipelines, but recent work shows pretrained models can perform generation in a unified text-to-text framework [ 13 , 18 ]. Our work extends these ideas to search, where catalog data creates opportunities for grounding generation in real-world entities. 

## 2.3 Relevance Label Generation 

Dense retrieval methods [ 10 , 14 ] require substantial training data with relevance labels, motivating automated labeling approaches. Benchmarks like BEIR [ 22 ] highlight the challenge of zero-shot generalization across domains. Recent work has explored using LLMs to evaluate and label data: Zheng et al . [27] demonstrated that strong LLMs can serve as reliable judges for evaluating model outputs, and Liu et al . [15] showed GPT-4 correlates well with human judgments. The importance of hard negatives for contrastive learning is well established [ 12 , 19 , 26 ], as easy negatives provide little training signal. Our approach combines two labeling strategies: contrastive generation that produces labels by construction, and Virtual Judge labeling using LLMs for broader coverage. We also develop techniques to control example difficulty at generation time. 

## 3 Synthetic Query Generation 

Our framework treats query generation as a text-to-text transforma-tion guided by three core requirements: (1) realism —queries must reflect authentic user linguistic patterns; (2) diversity —coverage across the full breadth of user intents and inventory attributes; and (3) scalability —efficient generation of millions of examples. Effective generation requires understanding how real travelers search. We identify three behavioral patterns: efficiency-oriented —users type terse queries like “beach house surfing” expecting the system to infer defaults; expectation-rich —queries like “romantic getaway” implicitly assume the system understands unstated pref-erences; and natural and imperfect —real queries contain typos, ab-breviations, and mixed languages. Critically, synthetic queries must avoid platform-specific terminology that users rarely type naturally.

## 3.1 Method Overview 

Figure 1 illustrates our contrastive query generation pipeline. The system combines two data sources with LLM processing to produce labeled synthetic queries: 

Input Data Sources. The pipeline draws from three primary sources: (1) Search Sessions —historical search and booking logs that capture real user behavior patterns, including which listings users viewed, engaged with, and ultimately booked; (2) Listing Features —structured catalog data including descriptions, titles, amenities, reviews, and other attributes that ground queries in factual platform content; (3) Seed Query Data —example queries from user research surveys (pre-launch) or small-traffic real user queries (post-launch) that provide authentic linguistic patterns for generation. 

Contrastive Pair Sampling. From search sessions with book-ings, we extract contrastive listing pairs: a booked listing (positive) and a sampled non-booked listing (negative) from the same search context. Both listings are then enriched with features from the cat-alog, creating listing pairs with features that capture meaningful attribute differences. 

LLM Processing. The core generation step combines three in-puts: (1) listing pairs with features providing factual grounding; (2) seed queries from user research providing linguistic patterns; and (3) prompting strategies controlling the realism-diversity trade-off. The LLM generates queries where one listing is more topical than the other. 

Output. The pipeline produces a database of generated queries, each associated with a labeled triplet (𝑞, 𝑙 +, 𝑙 − ) where the relevance relationship is known by construction. Algorithm 1 summarizes the generation process. 

## 3.2 Problem Formulation and Baselines 

Let 𝑃 true (𝑞 ) denote the true distribution of user queries—the proba-bility distribution over all possible natural language queries that real users would type when searching. This distribution is unob-served before launch: we cannot sample from it or measure its properties directly. Our objective is to generate a synthetic distri-bution 𝑃 synth (𝑞 ) that approximates the true distribution as closely as possible: 

min 𝐷 KL (𝑃 true (𝑞 )∥ 𝑃 synth (𝑞 )) (1) where 𝐷 KL denotes the Kullback-Leibler divergence, measuring how much 𝑃 synth differs from 𝑃 true in terms of information content. Minimizing this divergence means synthetic queries should be statistically indistinguishable from real queries in their linguistic patterns, attribute distributions, and intent coverage. We decompose query generation into two independent factors: 

• Template 𝑡 ∼ 𝑃 (𝑡 ): An abstract linguistic pattern capturing query structure—length, phrasing style, number of attributes mentioned, and constraint types. Templates are sampled from user research surveys representing how real travelers express their needs. 

• Entity 𝑒 ∼ 𝑃 (𝑒 ): A concrete listing providing factual at-tributes to ground the query—specific amenities, location features, property characteristics. Entities are sampled from the platform’s listing catalog. The synthetic distribution is then: 

𝑃 synth (𝑞 ) = E𝑡 ∼𝑃 (𝑡 ),𝑒 ∼𝑃 (𝑒 ) [𝑃 LLM (𝑞 | 𝑡, 𝑒 )] (2) where 𝑃 LLM (𝑞 | 𝑡, 𝑒 ) is the probability the LLM assigns to generat-ing query 𝑞 given template 𝑡 and entity 𝑒 . This decomposition en-ables independent improvement: refining 𝑃 (𝑡 ) using user research, expanding 𝑃 (𝑒 ) using catalog data, and improving 𝑃 LLM through prompt engineering. Naive LLM generation without platform knowledge (e.g., prompt-ing an LLM directly without Airbnb listing features) is obviously inadequate—it produces queries not grounded in actual inventory and cannot generate meaningful relevance labels. Therefore, for the remainder of this paper, we compare against a more competitive 

baseline : contrastive generation without seed data. This baseline uses listing pairs for factual grounding but lacks guidance on au-thentic linguistic patterns, producing overly verbose queries (e.g., “I am seeking accommodations with swimming pool facilities”) rather than terse natural queries (e.g., “pool near beach”). This comparison isolates the contribution of seed data: our approach combines con-trastive generation with seed queries from user research, producing queries that are both factually grounded and linguistically realistic. ## 3.3 Contrastive Generation 

Contrastive generation grounds synthetic queries in real platform features by using listing pairs sampled from actual user behavior. This ensures generated queries reference attributes that exist in the inventory rather than hallucinated features. 

Listing Pair Sampling. We sample pairs from real booking ses-sions to capture authentic user preference signals. Given a search context (location, dates, guests), we select a positive listing 𝑙 + (booked or highly engaged) and a negative listing 𝑙 − from the same search with less engagement. This ensures the listings share contextual relevance while differing in meaningful attributes. 

Listing Feature Enrichment. Each listing in the pair is en-riched with structured features included in the prompt. ## 3.5 Generation Quality Validation 

We validate synthetic query quality by comparing distributions against seed queries from user research, real user queries collected after launch, and a baseline (contrastive generation without seed data). Table 2 compares all four datasets. Figure 3 in Appendix visualizes the word count distributions. Real users are significantly more terse than synthetic queries—the median real query contains only 4 words (e.g., “pet friendly cabin”), compared to 6 words in our synthetic data. Real users show a right-skewed distribution with 33% using 1–2 words. Our seed-guided approach captures some of this terseness (6% short queries). 

Outperforming baseline: The baseline (contrastive generation without seed data) produces verbose queries—mean length 13.6 words with 97.6% exceeding 8 words—resulting in KL divergence of 4.95 vs. real users. Our seed-guided approach reduces this to 0.66, a 7.5 × improvement. The baseline’s verbosity stems from LLMs’ tendency to generate complete, grammatically elaborate sentences when unconstrained, while seed templates guide generation toward the terse patterns real users exhibit. 

Distribution comparisons: We measure distributional simi-larity using KL divergence on word count distributions. The KL divergence from real to seed (0.15) is lower than real to synthetic (0.66), which may seem surprising given that synthetic queries have a mean length (6.0) closer to real users (5.0) than seed queries do (6.6). However, KL divergence measures the shape of the dis-tribution, not just its center. Real users and seed queries share similar distributional characteristics: both exhibit high variance (𝜎 = 3.96 and 4.1), right-skewed distributions (mean > median), and spread-out query lengths (50–57% in the 3–8 word range). In contrast, synthetic queries are tightly concentrated ( 𝜎 = 2.5, 84% in 3–8 words) with a symmetric distribution. This concentration occurs because our generation process, guided by templates and listing features, produces consistently structured outputs. Despite this shape difference, synthetic data remains valuable: attribute cov-erage and semantic content—not query length distribution—drive ranking model performance.

## 4 Label Generation 

Training ranking models requires topicality labels —judgments of which listings better match a query. We develop two complementary approaches. 

## 4.1 Method Overview 

Contrastive Label Generation. Our first approach produces la-beled triplets (𝑞, 𝑙 +, 𝑙 − ) by construction—instead of generating la-bels for queries, we generate queries for labels. Given a listing pair, we prompt the LLM: “Generate a query where the first listing is more topical than the second.” This yields high-fidelity labels (the relationship is a deterministic input, not inferred) and a uni-fied pipeline (queries and labels in one step). However, generated queries may be overly specific to the listing pair, limiting diversity. 

Virtual Judge (VJ) Label Generation. Our second approach decouples query and label generation. We first generate queries using contrastive generation, then discard the original labels and re-evaluate against different listings using an LLM judge. The VJ eval-uates attribute matching, implicit intent, and relative comparison.

## 4.2 Considerations of LLM Evaluator 

A potential concern with using LLMs to evaluate LLM-generated content is self-preference bias : LLM judges may systematically favor outputs that match their own generation patterns. Liu et al . [15] found that GPT-4-based evaluators consistently assigned higher scores to GPT-3.5 generated summaries compared to human-written summaries, even when human judges preferred the human-written versions. They hypothesize this occurs because LLMs may share similar evaluation criteria during both generation and evaluation phases, leading to self-reinforcing preferences. This concern is particularly relevant for Virtual Judge labeling, where similar LLMs generate both queries and relevance judgments. Table 4 provides empirical evidence of this bias in our setting: when GPT models evaluate contrastive pairs generated by GPT, they achieve 98–99% accuracy—substantially higher than embedding models (76–79%, Table 5). This near-perfect agreement suggests LLMs recognize patterns in their own generations that may not reflect true topicality differences. We address this through several mitigations: (1) attribute ground-ing —judgments are based on explicit, verifiable listing attributes (amenities, location, property type) rather than stylistic preferences, reducing the influence of generation patterns; (2) human calibra-tion —we validate VJ labels against human judgments on a held-out sample, finding 87% agreement; (3) separation of concerns —we use contrastive labels (deterministic by construction) for training, re-serving VJ labels primarily for evaluation where coverage matters more than perfect fidelity; and (4) diverse model usage —when possi-ble, we use different model variants for generation vs. evaluation. While self-preference bias remains an open research question with implications for self-reinforcing LLM training loops, our eval-uation approach based on attribute grounding appears to mitigate its impact in our setting. The high LLM accuracy also validates that our contrastive generation produces consistent, well-formed examples—the concern is using LLM judgments as ground truth rather than as quality checks. In practice, we use contrastive la-bels for model training (where fidelity is critical) and VJ labels for evaluation (where coverage matters more).

## 4.3 Metrics Choice and Evaluation Results 

Although listing pairs are sampled from booking sessions, the la-bels we generate are topicality judgments—the prompt explicitly instructs the LLM to generate queries where one listing is more topically relevant than the other. This design choice reflects our cold-start focus: without sufficient booking data, we optimize for query-listing semantic relevance rather than conversion predic-tion. We use pairwise accuracy rather than NDCG (Normalized Discounted Cumulative Gain) because NDCG requires booking data to establish graded relevance, which is unavailable in cold-start. Pairwise accuracy is natural for our contrastive setup, which produces (𝑞, 𝑙 +, 𝑙 − ) triplets where the correct ranking is known by construction. Accuracy ranges from 0.5 (random guessing) to 1.0 (perfect), and we compare model performance on baseline (con-trastive without seed) versus our proposed approach. Transitioning to bookability-focused NDCG optimization is future work (Sec-tion 6.2). ## 5 Production Deployment 

We deploy the synthetic data framework in production at Airbnb, supporting retrieval and ranking evaluation. 

Pipeline Overview. The production pipeline refreshes daily, reading from historical search sessions and listing features that are also updated daily—ensuring generated queries are never static. Each day, the pipeline: (1) ingests fresh search sessions with book-ing logs and updated listing catalog features; (2) filters sessions with bookings to extract booked and non-booked listing pairs, en-riched with catalog attributes; (3) combines listing pairs with seed queries using an 80/20 mix of seed-guided (realistic) and variety (exploratory) prompts to generate approximately 10,000 synthetic queries with topicality labels; (4) analyzes generated queries against seed data and computes quality metrics (e.g., KL divergence, length distributions, attribute distributions) to monitor generation qual-ity; and (5) refreshes dashboards for ranking models trained and evaluated on this data. These quality metrics guide iteration on data generation parameters and prompt tuning, combining offline prototyping analysis with online deployment monitoring. 

From Cold Start to Warm Start. The framework supports con-tinuous refinement as the system matures. Cold start (pre-launch) :We use approximately 500 curated queries from user research sur-veys as seed templates—these tend toward verbose responses but provide essential signal about user intent patterns. Warm start (post-launch) : As real user traffic becomes available, we update seed data inputs with representative samples, generate analysis reports com-paring synthetic vs. real query distributions, and adjust generation strategies based on identified gaps. Each batch of real queries im-proves the next generation of synthetic data, gradually closing the distribution gap between synthetic and real user behavior. 

## 6 Discussion 6.1 When Synthetic Data Works 

Synthetic data proves most effective when: (1) applications have clear, well-defined evaluation metrics like pairwise ranking accu-racy; (2) teams need to iterate offline before real traffic exists; and (3) cold-start scenarios require bootstrap training signal. The key insight is that synthetic data need not be perfect—it needs only to be informative enough to guide development in productive directions. Models developed with synthetic data transfer reasonably well to real traffic, though some adaptation is typically needed post-launch. 

## 6.2 Limitations and Future Directions 

Our framework assumes independence between templates and en-tities, though in reality query structures likely correlate with at-tribute types. Future work could model 𝑃 (𝑒 |𝑡 ) to capture these correlations. Distribution shift between synthetic and real queries

is inherent—we view synthetic data as a bridge to real data rather than a permanent replacement. LLM limitations propagate into synthetic data; as foundation models improve, generation quality should improve correspondingly. 

From Topicality to Bookability. This paper focuses on topi-cality because cold-start precludes booking-based optimization. As real user data accumulates post-launch, several transitions become possible: (1) metric transition from pairwise accuracy to NDCG, using booking behavior to establish graded relevance; (2) data tran-sition where real user queries replace or augment seed data; and (3) label transition where click and booking signals supplement synthetic labels with ground-truth relevance. 

## 6.3 Practical Lessons Learned 

Our iterative development process yielded several practical insights. 

Start Simple. Our initial implementation used only listing titles and descriptions, with basic contrastive generation. While limited, this baseline enabled downstream development to begin—retrieval engineers could prototype embedding approaches, ranking teams could establish training pipelines. Teams that wait for perfect syn-thetic data may find themselves perpetually blocked. 

Difficulty Calibration. Our initial dataset showed uniformly high accuracy across all models. When all models achieve 90%+ accuracy, there is no signal to guide improvement. Controlled dif-ficulty tiers transformed our evaluation into a nuanced assess-ment that could distinguish incremental model improvements. We achieved this through prompt engineering techniques including leakage guardrails, difficulty buckets, and hierarchical sampling of listing pairs with controlled similarity. 

Production Scale Considerations. Model choice matters for cost and latency—we use smaller models for bulk generation and reserve larger models for quality validation. Feature selection re-quires careful truncation: we limit listing descriptions and select top amenities by relevance, balancing context richness against token costs. Prompt iterations are essential to match product requirements such as query length targets and attribute coverage. Daily refresh ensures generated queries are never static and enables continuous monitoring of generation quality. 

## 7 Conclusion 

This paper presents a framework for generating synthetic search queries and labels to address the cold-start challenge in natural language search. We combine contrastive listing pairs (ground-ing queries in real platform features) with seed queries from user research (ensuring realistic linguistic patterns), developing three prompt variants—seed controlled, seed freeform, and variety—to balance realism and diversity. For label generation, we introduce contrastive generation that produces labels by construction and Virtual Judge labeling for broader coverage. Experiments demonstrate the critical importance of seed data across multiple dimensions. For both query length and attribute distributions, our seed-guided approach significantly outperforms baselines—achieving a synergistic effect where the combination exceeds either component alone. Our approach also produces harder evaluation examples, providing discriminative signal for model improvement. The framework supports both retrieval and ranking evaluation, with a production pipeline that evolves from cold start to warm start as real user data becomes available. Several lessons emerge from our experience. Starting simple en-ables progress even with imperfect data. Difficulty calibration is essential—uniformly high accuracy provides no improvement sig-nal. Production scale requires balancing model cost, feature trunca-tion, and iterative prompt refinement. Future work includes transi-tioning from topicality-based accuracy to bookability-based NDCG as real user data becomes available, and establishing frameworks for graceful transition from synthetic to real user data post-launch.

## D Prompt Templates 

Our prompt system uses modular components that are composed into three main variants: Seed Controlled (template-based gener-ation with explicit three-step instructions), Seed Freeform (direct constraints with flexible attribute selection), and Variety (relaxed constraints with higher temperature for diversity). This modular ar-chitecture enables systematic A/B testing of individual components while maintaining consistency across variants. The following com-ponents are shared across all prompt variants, ensuring consistent behavior regardless of generation mode. 

Core Assumption. This establishes the contrastive generation objective—the LLM must generate a query that makes the specified relevance relationship true: 

Listing 1 is topically more relevant to the ideal query than Listing 2. Your job is to identify what search query would make this true. 

Platform Terms Blocklist. Real users don’t use internal plat-form terminology. This blocklist prevents the LLM from generating queries containing terms that would never appear in authentic user input: 

PLATFORM_TERMS = ["entire home", "private room", "superhost", "guest favorite", "badge", "airbnb", "listing", "instant book"] 

Context Deduplication Rules. Since production ranking mod-els receive both the query text and structured search context (lo-cation, dates, guests), generated queries should not redundantly repeat information already captured in the context: 

- Location: NO exact city/state names If location="Paris" -> DON’T use "Paris" OK: "near Eiffel Tower", "downtown" - Guests: NO exact counts If guests=4 -> DON’T say "for 4 people" OK: "family-friendly", "large group" - Dates: NO specific dates OK: "Christmas stay", "summer retreat" 

Consistency Enforcement. Generated queries must be consis-tent with the search context. A query mentioning “pet-friendly” for a search with no pets would create invalid training examples: 

If pets=0: DO NOT mention "pet-friendly" If children=0: DO NOT mention "family-friendly"
